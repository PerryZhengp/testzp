# 产品需求文档（PRD）

## 一、项目定位

### 概念

创建一款原创的浏览器解谜游戏，灵感来自于“不可能几何”和静谧探索。玩家控制一名无对白的旅行者，在浮空建筑中通过点击旋转塔楼、滑动桥梁、升降立柱以及利用透视错觉来连接道路，最终抵达关卡终点。为了避免版权问题，本项目不使用 **Monument Valley** 中的任何名字、角色、关卡或美术，而是专注于体验抽象：不可能几何、光学错觉、通过操纵建筑显露路径、冥想式的解谜体验。

### 核心关键词

不可能几何、静谧探索、低挫败感、点按交互、章节式关卡、视觉即提示。

### 推荐目标

第一阶段实现一个最小可行产品（MVP）：

- 1 个章节，总计 8 个关卡，时长 30–60 分钟。
- 提供 4 种核心机制：旋转塔楼、滑动桥梁、升降立柱、透视对齐路径。
- 支持本地自动存档和设置页面，无后端依赖。

## 二、设计原则

1. **看懂优先于难住玩家**：玩家第一眼要知道哪里能走、哪里能点、目标在哪。
2. **每关只教一个新知识点**：新机制出现时，不要同时引入第二个复杂变量。
3. **点击就是全部核心操作**：保留点击/点按移动和点击机关，不做自由走位或复杂动作输入。
4. **低惩罚、低焦虑**：没有生命值、倒计时或敌人；失败成本只是一键重置本关。
5. **每关必须有一个“哇”时刻**：例如视角一对齐，道路突然贯通；或建筑旋转后出现隐藏楼梯。
6. **每个画面都像一张海报**：场景构图、配色、动画节奏都要服务于观感。

## 三、MVP 范围

### 必做功能

- 开场页、主菜单、章节/关卡选择。
- 关卡加载与游玩：角色点击移动、机关交互、完成判定。
- 自动存档：记录解锁关卡和设置项。
- 暂停、重新开始、返回选关。
- 设置页面：音量、语言、低动态效果。
- 简易提示系统：玩家停滞 45–60 秒后给弱提示。

### 首版核心机制

1. **旋转塔楼**：建筑的某些部分以 90° 步进旋转，改变楼梯或桥的朝向。
2. **滑动桥梁**：沿固定轨道水平移动，用于连通断路。
3. **升降立柱**：抬升或降低平台高度，形成新的通路或阻断旧路。
4. **透视对齐**：两个物理上不相连的平台，在特定视角下投影重合时形成可通行的错视路径。

### 不在首版范围内的内容

- 自由摄像机、多人模式、战斗系统、随机生成、后台服务。
- 复杂剧情或对话树、传送门、光束折射、可移动 NPC。
- 在线存档、关卡编辑器或程序化关卡。

## 四、核心玩法需求

### 4.1 角色移动

- 玩家点击可达节点，角色自动沿寻路路径移动到该节点。
- 禁止自由 WASD 走位；点击不可达位置时给予轻微反馈（抖动、闪烁或音效）。
- 角色移动速度恒定，转身与上下楼梯有平滑动画。
- 角色到达终点节点时触发关卡完成。

### 4.2 机关交互

- 所有可交互物必须有明显提示：高亮描边、呼吸动画或悬停发光。
- 点击机关后，先播放机关动画，再刷新行走图。
- 动画执行期间锁定输入，不允许队列化多个操作。
- 机关的所有状态必须可序列化保存，但首版不在关内保存状态。

### 4.3 摄像机

- 固定正交摄像机；玩家不能自由旋转或拖拽视角。
- 摄像机只允许通过脚本进行轻微推拉或切换，用于引导视线或过渡。
- 透视错觉建立在受控视角之上；每关建议只有一个主视角，最多一次转场。

### 4.4 关卡目标

- 主目标为抵达终点祭坛/门/光塔，终点最好在关卡开始后 3–5 秒内可见。
- 首版不做收集品或次要目标。

### 4.5 提示系统

- 不写大量文字教程；首次出现新机关时用镜头聚焦和唯一通路教会玩家。
- 玩家停滞 45–60 秒后提供弱提示，不直接给出答案。

### 4.6 存档与进度

- 本地自动存档，只保存已解锁/完成的关卡和设置项。
- 刷新页面后能够从最近一次进入的关卡初始状态继续游戏。
- 首版不保存关内瞬时机关状态或玩家位置。

## 五、关卡设计要求

### 5.1 关卡结构模板

每关按以下四步设计：

1. **展示**：让玩家看懂空间结构和目标位置。
2. **教学**：引入一个新的核心机关，保证唯一解。
3. **组合**：把新机制与之前的知识点组合，增加复杂度但不过度叠加。
4. **回报**：通过意想不到的视角连通或结构变化带来惊喜感。

### 5.2 关卡长度与节奏

- 前三关：每关 2–4 分钟，逐步建立基本概念。
- 中段关卡：每关 4–6 分钟，开始组合两个机制。
- 终局关卡：6–8 分钟，使用最多 3 种机制并制造大规模场景变化。

### 5.3 设计约束

- 早期关卡最多涉及 2 类机制；后期最多 3 类。
- 每关可交互物数量控制在 2–6 个。
- 不允许像素级寻找热点；所有可交互元素必须有视觉提示。
- 不允许出现无解死局；任何状态要么可回退，要么可一键重置。

### 5.4 建议难度曲线（8 关）

| 关卡 | 主要教学内容 |
|---|---|
| **L1** | 移动与终点，教会玩家点击节点移动 |
| **L2** | 引入旋转塔楼 |
| **L3** | 引入滑动桥梁 |
| **L4** | 旋转 + 滑动组合 |
| **L5** | 引入透视对齐路径 |
| **L6** | 升降立柱 + 透视对齐 |
| **L7** | 旋转、滑动、升降三机制组合 |
| **L8** | 大型终局关，强调景观变化与“哇”时刻 |

## 六、美术与音频指引

### 6.1 美术方向

原创世界观，避免使用 Monument Valley 中的任何识别性元素。可以围绕以下主题设计：

- 浮空天文台
- 陶瓷遗迹
- 纸雕修道院
- 潮汐钟塔

### 6.2 视觉规则

- 使用简洁几何体：柱、台阶、拱门、桥、轮盘等。
- 一章只使用 3–5 种主色，可交互物的饱和度略高于背景。
- 贴图克制，靠形体、光影和配色构建风格。
- 避免直接沿用 Monument Valley 的角色轮廓、帽型、披风比例等。
- 不使用乌鸦人或面具守卫这类明显可识别的敌对元素。

### 6.3 动画规则

- 机关动画时长 300–900 ms，缓动柔和而优雅，不做硬切。
- 角色移动、转身和上下楼梯的节奏统一连贯。
- 场景变化应“优雅”，而非“机械暴力”；错视连通时需要有惊喜感。

### 6.4 音频规则

- 环境音轻盈、空灵且可循环，营造冥想氛围。
- 机关音效短促且有质感，完成关卡时有温和向上的音乐动机。
- 避免使用强鼓点或高压紧张配乐。

## 七、技术实现指南

### 7.1 总体思路

- 使用 **TypeScript** + **Three.js** 搭建渲染层，固定正交摄像机。
- 使用 **Vite** 作为构建工具，不依赖 React；UI 层使用 DOM 元素覆盖在 Canvas 上。
- 所有关卡数据使用 JSON/TS 描述，实现数据驱动的玩法。
- 区分 **Scene Graph**（场景图）与 **Walk Graph**（可行走图）：渲染与逻辑解耦。
- 使用本地存储保存解锁关卡和设置，无后端依赖。

### 7.2 关键数据模型

- **NodeDef**：站立点，具有 `id` 与相对坐标，挂靠在某个可变换的锚点上。
- **EdgeDef**：连接两个节点的边，分为 `real`（永久）、`conditional`（机关驱动）和 `illusory`（错视）。条件使用结构化逻辑描述，而非字符串表达式。
- **InteractableDef**：可交互机关，具有有限离散状态，每个状态对应一个值和动画时长；仅在动画完成后改变状态。
- **CameraDef**：为每个关卡定义固定的缩放、偏航角和俯仰角。
- **LevelDef**：聚合节点、边、机关、摄像机、出生点和终点等信息。

### 7.3 错视连边的实现

错视连边是游戏的灵魂。每条 `illusory` 边都明确指定 `from` 和 `to` 两个节点以及一个 NDC 阈值。系统计算这两个节点的世界坐标投影到标准化设备坐标（NDC）后的 2D 距离，只有当投影距离小于阈值且满足其他条件时，该边才被视为可行走。这样可以避免误连接任意临近的节点，并保证在不同分辨率下谜题逻辑一致。

### 7.4 系统模块（建议）

- `SceneManager`：管理场景图、模型和灯光。
- `LevelLoader`：解析 `LevelDef` 并构建场景和行走图。
- `InputManager`：统一处理鼠标/触屏输入，对命中代理进行射线检测。
- `InteractionSystem`：管理机关点击、动画播放和状态变更。
- `GraphSystem`：根据当前机关状态和错视条件重建可行走图。
- `MovementSystem`：执行 A* 寻路并驱动角色移动动画。
- `CameraController`：按照关卡设置控制固定摄像机，支持过渡动画。
- `SaveService`：读写本地存档，仅保存元进度和设置。
- `AudioService`：播放背景音乐、机关音效和 UI 音效。
- `GameStateMachine`：全局和游玩子状态机，串行化输入，避免状态竞争。

### 7.5 性能与兼容性

- 桌面端目标 60 FPS；移动端可接受 30–60 FPS。
- 首屏进入可交互状态控制在 3–5 秒；重开当前关卡在 1 秒内。
- 首版优先支持桌面 Chrome/Edge/Safari 和 iPad Safari；移动端适配为次优目标。

## 八、验收标准

关卡和系统完成后，应满足以下条件：

- 玩家无需阅读说明即可完成前 3 关；所有可交互物有明确反馈。
- 点击不可达节点时提供轻微反馈，角色不会穿模或走入无路状态。
- 重置关卡可在任意状态下恢复到初始布局，不存在无解死局。
- 错视连接只有在指定条件下生效，不会误连其他节点。
- 刷新页面后仍能保留已解锁的进度；只需鼠标即可完成全部操作。
- 触屏点击区域足够大，移动端不会频繁误触；完成关卡后有清晰但不突兀的反馈。

## 九、后续指引

在实现阶段，可以参考本文档末尾“技术规格”中的闭环约束和模块划分。为提高实施效率，建议先按照提示词顺序交给生成式编码助手：

1. 创建项目骨架和目录结构；实现全局状态机和渲染循环。
2. 实现数据驱动的关卡系统、节点/边模型和点击移动。
3. 加入旋转、滑动、升降三类机关，使用离散状态驱动动画与逻辑。
4. 实现透视对齐连边，基于 NDC 计算启用/禁用。
5. 完成流程闭环：关卡完成、解锁链、存档、设置、暂停和提示系统。
6. 开发关卡校验器和单元测试，确保每个关卡可解且无死局。

更多细节及技术约束见 `TECH_SPEC.md`。
